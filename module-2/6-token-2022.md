# 6. Token-2022 (newer token standard)

## TL;DR

**Token-2022** = newer Solana token standard. Same basics як legacy SPL Token + "extensions" — opt-in features per mint. Important: not interchangeable з legacy tokens. New use cases enabled (transfer fees, confidential transfers, RWAs, soul-bound NFTs).

## Two token programs co-exist

```
SPL Token (legacy):      TokenkegQfeZyiNwAJbNbGKPFXCWuBvf9Ss623VQ5DA
Token-2022 (newer):      TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb
```

Both run on mainnet. Tokens belong до one or other (mint's `owner` field). USDC, JTO, JitoSOL = legacy. Many newer tokens (especially RWAs, niche use cases) = Token-2022.

## Extension types

Mint можуть enable any subset extensions at creation time:

### Transfer fees

Per-transfer % goes до specific authority. Use cases:
- Royalty payments (NFT-style)
- Protocol revenue
- Anti-spam mechanism

### Confidential transfers

Use zero-knowledge proofs for hiding amounts. Privacy-preserving. Compliance considerations (requires special auditor key).

### Interest-bearing

Token balance auto-grows based on configured interest rate. No manual claim. Use:
- Yield-bearing stablecoins
- Bond representations

### Non-transferable (soul-bound)

Once received, cannot transfer. Use:
- Achievements / badges
- Identity verification
- Membership tokens

### Default frozen state

New token accounts frozen until explicit unfreeze. Use:
- Whitelisted token (only approved holders)
- KYC-gated tokens

### Permanent delegate

Mint authority always can move/burn tokens. Use:
- Compliance freezing (regulator requires)
- Stablecoin issuer redemption mechanism

### Transfer hooks

Custom program runs on each transfer. Validates або triggers side effects:
- Compliance checks per transfer
- Triggering external programs
- Logging

### Metadata pointer

Embeds metadata reference directly у mint. Replaces external metadata accounts (Metaplex pattern). Simpler NFT setup.

## Use case examples

### RWA (Real World Assets) tokens

E.g., tokenized treasury bills. Need:
- Permanent delegate (regulator can freeze/burn)
- Transfer hooks (KYC/sanctions checks)
- Interest-bearing (yield accrual)

→ Token-2022 з всіма цими extensions.

### Privacy stablecoin

E.g., regulated privacy token. Need:
- Confidential transfers
- Permanent delegate (for compliance)

→ Token-2022 з confidential transfers + permanent delegate.

### Membership token

E.g., DAO governance non-transferable token. Need:
- Non-transferable (soul-bound)

→ Token-2022 з non-transferable extension.

### Standard fungible (USDC-style)

Just basic transfer + freeze. No extensions needed.

→ Either Token-2022 (без extensions) OR legacy SPL Token (likely choice для standard tokens).

## CLI usage

`spl-token` CLI supports both. Specify program:

```bash
# Legacy (default)
spl-token create-token

# Token-2022
spl-token create-token --program-id TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb
```

For accounts:

```bash
# Show all token balances (auto-detects both programs)
spl-token accounts

# Create ATA для Token-2022 mint
spl-token create-account <TOKEN_2022_MINT> --program-id TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb
```

## Verify which program owns mint

```bash
solana account <MINT_ADDRESS> | grep Owner
```

- `Owner: TokenkegQfeZyiNwAJbNbGKPFXCWuBvf9Ss623VQ5DA` → legacy SPL Token
- `Owner: TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb` → Token-2022

## Connect to your work

Як validator operator, ти не creates tokens — but може receive them (rewards, treasury holdings):

- **Jito tips**: native SOL (no token program needed)
- **DZ rewards**: likely Token-2022 (modern protocol)
- **JitoSOL holdings**: legacy SPL Token
- **USDC у treasury**: legacy SPL Token

When configuring `--rewards-token-owner` (DZ context з 2026-06-08), the token program matters:
- Owner wallet still standard SOL wallet (System Program)
- ATA derivation differs slightly between programs (different program in seeds)
- DZ tooling handles this internally

## Migration considerations

Legacy → Token-2022 migration NOT auto. Users must:
1. Burn legacy tokens
2. Receive new Token-2022 tokens through bridge contract
3. Update wallet handling

Major existing tokens (USDC, JTO, etc.) unlikely migrate due to ecosystem inertia (DEXes, dApps integrated з legacy).

New token launches increasingly choose Token-2022 for flexibility.
